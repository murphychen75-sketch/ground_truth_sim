# ra223f_ws — RA223F 雷达 ROS2 驱动使用说明

本文档说明 **ra223f_ws** 工作空间的结构、编译、运行、网络通信与话题接口。工程包含两个包：`ra223f_pkg`（核心节点）与 `launch_pkg`（启动入口）。

---

## 一、工程概述

### 1.1 功能简介

- **UDP 组播接收**：从雷达组播地址接收点云/目标原始报文（0xAA80/0xAA81、0xAA90/0xAA91），并解析可选状态报文 **0xAA8B**（IP 装订结果日志）。
- **ROS2 发布**：将点云/目标发布为 `sensor_msgs/msg/PointCloud2` 与 `visualization_msgs/msg/MarkerArray`，便于 RViz 与下游模块消费。
- **UDP 单播注入（0xAA31）**：综合 **里程计**、**车速**、**IMU** 等话题缓存，在任一路话题回调时尝试组一包注入数据并单播到雷达（详见下文「注入与话题」）。

### 1.2 工作空间结构

```
ra223f_ws/
├── src/
│   ├── ra223f_pkg/
│   │   ├── src/
│   │   │   ├── ra223f_node.cpp

│       ├── launch/launch.py

└── 使用说明.md
```

### 1.3 依赖说明

| 包名 | 主要依赖 |
|------|----------|
| `ra223f_pkg` | rclcpp, sensor_msgs, nav_msgs, geometry_msgs, visualization_msgs |
| `launch_pkg` | launch, launch_ros |

## 二、环境与编译

### 2.1 环境要求

- **ROS2**：建议 Humble 或更高版本（`ament_cmake`）。
- **操作系统**：`udp_interface.*` 使用 POSIX/Linux 风格 socket（`sys/socket.h`、`IP_ADD_MEMBERSHIP` 等），建议在 **Linux** 或 **WSL** 下编译运行；原生 Windows 需自行替换为 Winsock 或交叉编译环境。

### 2.2 编译

在工作空间根目录执行：

```bash
cd ra223f_ws
source /opt/ros/<distro>/setup.bash
colcon build --packages-select ra223f_pkg launch_pkg
source install/setup.bash
```

## 三、运行方式

### 3.1 一键启动（推荐）

```bash
source install/setup.bash
ros2 launch launch_pkg launch.py
```

### 3.2 单独运行节点

```bash
source install/setup.bash
ros2 run ra223f_pkg ra223f_node
```

### 3.3 运行模式 `run_mode`（双雷达示例）

`main()` 内预留了同一进程内 **节点 A / 节点 B** 的创建逻辑。默认启用 **A**；**B** 段在源码中多为注释，需按现场 IP/端口解注后编译。

可通过命令行选择模式（不区分大小写）：

| 取值 | 含义 |
|------|------|
| `both`（默认） | 启用 A，且当 B 段代码已解注时启用 B |
| `a` | 仅启用 A |
| `b` | 仅启用 B（需 B 段已实现） |

**注意**：`run_mode` 是在 `rclcpp::init()` **之后**从 `argv` 解析的。`--ros-args -p run_mode:=...` 通常会被 ROS 提前消费掉，**不一定**能传到 `parse_run_mode`。推荐使用 **`--` 传给可执行文件** 的自定义参数：

```bash
ros2 run ra223f_pkg ra223f_node -- --ra223f-run-mode a
# 或
ros2 run ra223f_pkg ra223f_node -- --ra223f-run-mode=both
```

源码也支持在未被过滤的 `argv` 中出现 `run_mode:=a` 等形式；若与你的启动方式冲突，优先使用 `--ra223f-run-mode`。

无效取值将回退为 `both`，并在控制台提示。

### 3.4 通过 launch / 命令行覆盖 ROS 参数

`launch_pkg/launch/launch.py` 未写死参数，可在启动时追加 `--ros-args -p ...`，例如：

```bash
ros2 launch launch_pkg launch.py --ros-args -p odom_topic:=/odom -p multicast_interface_ip:=192.168.1.100
```

## 四、网络与通信（重要）

### 4.1 组播接收与单播发送是否可同时使用？

可以。工程内同时维护：

- **组播接收 socket**：接收雷达数据；
- **单播发送 socket**：发送 0xAA31 注入包。

二者独立，互不冲突（需网络可达、端口未被占用）。

### 4.2 组播/单播端点（编译期：`main()`）

雷达 **组播接收** 与 **单播注入目标** 的 IP/端口在 `ra223f_pkg/src/ra223f_node.cpp` 的 `main()` 中按节点配置，例如节点 A：

- **组播**：`multicast_a`（默认示例 `224.0.2.3:42104`）
- **单播**：`unicast_a`（默认示例 `10.13.2.115:42404`）

修改后需重新 `colcon build`。

单播客户端会从 `local_port_start`～`local_port_end`（构造默认 `50000`～`65535`）扫描本机绑定端口，直到 `bind` 成功。

### 4.3 组播网卡（运行期：ROS 参数）

多网卡或组播加入失败（如 ENODEV）时，可指定加入组播所使用的 **本机网卡 IPv4**：

| 参数名 | 类型 | 默认 | 说明 |
|--------|------|------|------|
| `multicast_interface_ip` | string | 与构造参数一致（`main()` 传入多为 `0.0.0.0`） | `0.0.0.0` 或空表示由系统选网卡；否则为 `setsockopt(IP_ADD_MEMBERSHIP)` 的接口地址 |

## 五、注入（0xAA31）与订阅话题

节点使用 **多线程执行器**；在 **Odometry**、**TwistStamped（车速）**、**Imu** 任一路回调中都会调用合成逻辑，向雷达发送一包 0xAA31（单播未初始化则跳过并告警一次）。

### 5.1 订阅与 QoS

| 话题（默认名） | 消息类型 | 说明 |
|----------------|----------|------|
| `odom_topic` → 默认 `/mavros/local_position/odom` | `nav_msgs/msg/Odometry` | **Best Effort** QoS，便于与 MAVROS 等发布端对齐 |
| `vehicle_speed_topic` → 默认 `/VCU_status/vehicle_feedback` | `geometry_msgs/msg/TwistStamped` | `twist.linear.x` 单位为 **km/h**，节点内 **÷3.6** 转为 m/s |
| `imu_topic` → 默认 `/imu/data` | `sensor_msgs/msg/Imu` | 使用 `angular_velocity.z`（rad/s）作为偏航角速度；`linear_acceleration.x` 会缓存，**当前 0xAA31 包未写入纵向加速度** |

默认里程计话题为 **`/mavros/local_position/odom`**，不是 `/odom`。若你的系统发布在其它名字上，请用参数 `odom_topic` 覆盖。

### 5.2 运行期可改参数一览

| 参数名 | 类型 | 默认值（与源码一致） | 说明 |
|--------|------|----------------------|------|
| `odom_topic` | string | `/mavros/local_position/odom` | 里程计话题名 |
| `vehicle_speed_topic` | string | `/VCU_status/vehicle_feedback` | 车速 `TwistStamped` |
| `vehicle_speed_is_signed_longitudinal` | bool | `false` | 无 odom 时是否把车速当作带符号纵向速度（倒车为负）推断挡位；默认按非符号车速处理 |
| `imu_topic` | string | `/imu/data` | IMU 话题名 |

### 5.3 合成规则（便于联调）

- **车速**：优先使用 `motion_mutex` 中来自 `TwistStamped` 的最新值（m/s）；若本次合成路径上 **刚收到一帧 Odometry**（`odom_callback` 置位并消费），则用 \( \sqrt{v_x^2+v_y^2} \) 覆盖车速，并用 `twist.linear.x` 的符号推断 `move_direction`（倒挡/前进等，写入协议字段）。
- **偏航角速度**：默认用 IMU 缓存的 `angular.z`；若本周期以 Odometry 为准，则用 `twist.angular.z`。
- **发往雷达的速度幅值**：`send_inject_packet_values` 内对速度取 **`fabs`** 后再按协议缩放（整数×0.01 m/s 等），与源码保持一致。

## 六、发布话题与坐标系

所有点云与 Marker 的 **`header.frame_id` 均为 `ra223f_radar`**（不再使用 `map`）。在 RViz 中将 **Fixed Frame** 设为 `ra223f_radar`，或在上游 TF 中将其连到你的地图/车体坐标系。

### 6.1 默认话题名（可通过构造函数/`main()` 修改）

| 话题名 | 类型 | 说明 |
|--------|------|------|
| `/ra223f/pointcloud_topic` | `sensor_msgs/msg/PointCloud2` | 雷达点云 |
| `/ra223f/objectList_topic` | `sensor_msgs/msg/PointCloud2` | 目标列表（每目标一个点），字段见下表 |
| `/ra223f/object_marker` | `visualization_msgs/msg/MarkerArray` | 目标包围盒（`CUBE`），与 object 列表同一时间戳 |

双节点模式下 B 段解注后可使用带前缀的话题名（以源码为准）。

### 6.2 点云 `PointCloud2` 字段（`/ra223f/pointcloud_topic`）

每点 **14×float32**，`point_step = 56`：

| 字段名 | 含义 |
|--------|------|
| x, y, z | 由 range、方位角、俯仰角换算的笛卡尔坐标（m） |
| range | 径向距离（m） |
| amb_rangerate | 模糊径向速度（m/s） |
| unamb_rangerate | 非模糊径向速度（m/s） |
| rangerate | 与 `unamb_rangerate` 相同（兼容旧字段） |
| azimuth_deg, elevation_deg | 方位角、俯仰角（度） |
| snr, rcs, confidence | 信噪比、RCS、置信度 |
| unamb_rangeratemask | 解模糊标志（uint8 以 float 发布） |
| snr_azi | 方位维信噪比 |

时间戳来自报文内 **PTP**（秒+纳秒），写入 `header.stamp`。

### 6.3 目标 `PointCloud2` 字段（`/ra223f/objectList_topic`）

每目标 **16×float32**，`point_step = 64`：中心 `x,y,z`，尺寸 `width,length,height`，`xvel_abs,yvel_abs,xacc_abs,yacc_abs,heading_angle`，`classify_type,classify_prob,objmotion_status,obstacle_prob,track_id`（整数类型以 float 存储以便在 RViz 中查看）。

## 八、协议简述（开发参考）

- **点云**：0xAA80（帧头）+ 多条 0xAA81（点数据），每点 **20** 字节。
- **目标**：0xAA90（列表头）+ 多条 0xAA91（目标），每目标 **43** 字节。
- **注入**：0xAA31（**32** 字节），含 yawRate、车速、挡位/方向等；校验为 `data_buf[0..30]` 逐字节和取 uint8 写入 `data_buf[31]`。
- **IP 装订反馈**：0xAA8B（长度 8 时解析），在日志中输出装订成功/失败/复位等中文提示。
- **时间**：点云头中含 PTP 时间，用于填充本帧点云及随后目标发布的 `header.stamp`。

更详细的字节偏移与比例系数见 `ra223f_node.cpp` 中 `switch(messageType)` 各分支。

## 九、常见问题

1. **组播收不到数据**  
   确认雷达向组播地址/端口发包；检查交换机 IGMP/防火墙；多网卡时设置 `multicast_interface_ip`；可用 Wireshark/tcpdump 验证。

2. **单播注入无效（0xAA31）**  
   确认 `unicast` IP/端口与雷达注入口一致；确认至少一路订阅话题有数据（Odom / 车速 / IMU）；确认本机端口段内 bind 成功（日志会打印本地端口）。

3. **有车速和 IMU 但没有里程计**  
   仍会在 `TwistStamped` 或 `Imu` 回调时发送注入包；若需与车体坐标严格一致，请发布 `nav_msgs/Odometry` 或把 `odom_topic` 指到实际里程计话题。

4. **Odometry 订阅不到**  
   默认话题为 `/mavros/local_position/odom` 且为 **Best Effort**；若发布端为 Reliable，需改源码 QoS 或使用桥接；也可用 `odom_topic` 指到匹配的话题名。

5. **Windows**  
   当前 UDP 实现面向 Linux/POSIX；建议在 Linux/WSL 下使用，或自行移植 Winsock。
